## Prerequisites

- **Node.js 20 – 26**
- **Salesforce CLI** (`sf`)
- A **Salesforce DX project** with a `sfdx-project.json` file at the root. The plugin uses this file to discover your package directories.
- The plugin searches for Apex files only inside folders named `classes`, `triggers`, or `testSuites` within your declared package directories. Non-standard folder names are not scanned.

## Install

```sh
sf plugins install apextestlist
```

## Usage

This tool identifies Apex tests using one of three Apex annotations, or a separate metadata filter. Annotations are read from `.cls` and `.trigger` files.

### 1. `@Tests:` (Custom Comment Annotation)

Add a comment with the `@Tests:` prefix to a class or trigger to declare which tests cover it. Multiple tests can be separated by commas, spaces, or both. Multiple `@Tests:` lines in the same file are merged.

```java
// @Tests: SampleTest, SuperSampleTest
// @Tests: IntegrationTest
public class Sample {
  // ...
}
```

```java
// @Tests: SampleTriggerTest
trigger SampleTrigger on Account (before insert) {
  // ...
}
```

When this file is included in a manifest (via `--manifest`), `SampleTest`, `SuperSampleTest`, and `IntegrationTest` will be added to the test list.

### 2. `@TestSuites:` (Custom Comment Annotation)

Add a comment with the `@TestSuites:` prefix to declare test suites whose members should be run. Multiple suites can be separated by commas or spaces.

```java
// @TestSuites: SampleSuite SampleSuite2
public class Sample {
  // ...
}
```

The plugin resolves each suite to its member classes via the `.testSuite-meta.xml` file in your `testSuites` folder.

A file can combine both annotations:

```java
// @Tests: DirectTest
// @TestSuites: SampleSuite
public class Sample {
  // ...
}
```

### 3. `@isTest` (Apex Annotation)

The tool detects all Apex classes marked with the [`@isTest`](https://developer.salesforce.com/docs/atlas.en-us.apexcode.meta/apexcode/apex_classes_annotation_isTest.htm) annotation and includes them automatically — no extra comment needed.

```java
@isTest
private class SampleTest {
  // Included automatically
}
```

### 4. `.test-dependencies.yml` (Centralized Metadata Filter)

Instead of annotating individual classes, you can map metadata members to test classes in a single YAML file. This is designed to work with manifest files generated by [`sfdx-git-delta`](https://github.com/scolladon/sfdx-git-delta).

1. Create a `.test-dependencies.yml` file in the same directory as your `sfdx-project.json`.
2. Map test classes to metadata entries using `MetadataType:MetadataMember` format. Use `*` as a wildcard to match all children under a metadata parent.

```yml
MyFooTest:
  - CustomField:Account__c.field__c
  - Flow:OpportunityFlowX
  - ValidationRule:Account.RequiredEmail
  - CustomField:Opportunity__c.*

MyBarTest:
  - CustomObject:Lead__c
  - Flow:LeadFlowY
```

3. Run the command with `--filter-by-metadata` and `--manifest`:

```sh
sf apextests list --manifest package/package.xml --filter-by-metadata
```

Only test classes whose declared dependencies match metadata in the manifest will be included in the output.

> **Note:** The plugin will throw an error if `.test-dependencies.yml` exists but does not match the expected `Record<string, string[]>` structure.

## Running the Tool

Run the command from the root of any Salesforce DX project:

```sh
sf apextests list
```

### Output Formats

Use `--format` (`-f`) to control the output shape:

| Format         | Flag            | Output                          | Use with                                                 |
| -------------- | --------------- | ------------------------------- | -------------------------------------------------------- |
| `sf` (default) | `--format sf`   | `--tests A --tests B --tests C` | `sf project deploy start`                                |
| `sfdx`         | `--format sfdx` | `--tests A B C`                 | `sfdx force:source:deploy --testlevel RunSpecifiedTests` |
| `csv`          | `--format csv`  | `A,B,C`                         | Custom scripts, spreadsheets                             |

**Example — `sf` format (default):**

```sh
$ sf apextests list --format sf
--tests SampleTest --tests SuperSampleTest --tests Sample2Test --tests SuperSample2Test
```

**Example — deploy using command substitution:**

```sh
sf project deploy start $(sf apextests list)
# Expands to:
sf project deploy start --tests SampleTest --tests SuperSampleTest --tests Sample2Test
```

If no test methods are found, the command output will be empty and a warning is printed:

```
Warning: No test methods found
```

### Handling Missing Tests

By default, the tool does **not** verify that discovered test class names exist on disk. To enable that check:

```sh
sf apextests list --ignore-missing-tests
```

Tests not found in any package directory are printed as warnings and excluded from the output.

### Handling Missing Annotations

By default, the tool warns for each scanned file that has no annotation:

```
Warning: File "Sample.cls" does not contain @tests, @testsuites, or @istest annotations
```

Suppress these warnings with:

```sh
sf apextests list --no-warnings
```

## CI/CD Integration

A typical pipeline using [`sfdx-git-delta`](https://github.com/scolladon/sfdx-git-delta) to generate a delta manifest and this plugin to resolve tests:

```yaml
# GitHub Actions example
- name: Install SF plugins
  run: |
    echo y | sf plugins install sfdx-git-delta
    echo y | sf plugins install apextestlist

- name: Generate delta manifest
  run: |
    sf sgd source delta --to HEAD --from HEAD~1 --output-dir .

- name: Deploy with targeted tests
  run: |
    sf project deploy start \
      --manifest package/package.xml \
      --test-level RunSpecifiedTests \
      $(sf apextests list --manifest package/package.xml)
```

Using the metadata filter approach with `.test-dependencies.yml`:

```yaml
- name: Install SF plugins
  run: |
    echo y | sf plugins install sfdx-git-delta
    echo y | sf plugins install apextestlist

- name: Deploy with metadata-driven test selection
  run: |
    sf sgd source delta --to HEAD --from HEAD~1 --output-dir .
    sf project deploy start \
      --manifest package/package.xml \
      --test-level RunSpecifiedTests \
      $(sf apextests list --manifest package/package.xml --filter-by-metadata)
```

## Command Reference

```
USAGE
  $ sf apextests list -f <value> -x <value> -s -n -d <value> -m [--json]

FLAGS
  -f, --format=<value>            Output format. Available options:
                                    - `sf` (default): one --tests flag per class
                                    - `sfdx`: single --tests flag, space-separated
                                    - `csv`: comma-separated values
  -x, --manifest=<value>          Manifest XML file (package.xml) to filter test annotations.
  -s, --ignore-missing-tests      [default: false] Exclude test methods not found in any local package directory.
  -n, --no-warnings               [default: false] Suppress warnings for files missing annotations.
  -d, --ignore-package-directory  Ignore a package directory when looking for test annotations.
                                  Should match the path declared in sfdx-project.json.
                                  Can be declared multiple times.
  -m, --filter-by-metadata        [default: false] When used with --manifest, select tests based on
                                  metadata changes declared in .test-dependencies.yml rather than
                                  Apex annotations.

GLOBAL FLAGS
  --json  Format output as JSON.

EXAMPLES
  List all test annotations found in package directories in Salesforce CLI format:

    $ sf apextests list --format sf

  List all test annotations in CSV format:

    $ sf apextests list --format csv

  List test annotations only for Apex classes/triggers in a manifest file:

    $ sf apextests list --format sf --manifest package.xml

  List test annotations only if they exist in the package directories:

    $ sf apextests list --format sf --ignore-missing-tests

  Exclude annotations found in the "force-app" directory:

    $ sf apextests list -d "force-app"

  List test annotations without printing warnings for each file missing annotations:

    $ sf apextests list -n

  Use metadata dependencies file to select tests based on changed metadata:

    $ sf apextests list --manifest package.xml --filter-by-metadata
```
